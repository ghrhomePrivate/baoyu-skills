# ohmsha

Ohmsha 预设 - 带视觉隐喻的教育漫画

## 基础配置

| 维度 | 值 |
|-----------|-------|
| 画风 | manga |
| 基调 | neutral |
| 布局 | webtoon（默认） |

等价于：`--art manga --tone neutral`

## 独特规则

此预设包含超出画风+基调组合的特殊规则。当使用 `--style ohmsha` 时，必须应用以下所有规则。

### 视觉隐喻要求（关键）

每个技术概念必须以隐喻方式可视化：

| 概念类型 | 可视化方式 |
|-------------|----------------------|
| 算法 | 演示过程的道具/机器 |
| 数据结构 | 角色可以进入/探索的物理空间 |
| 数学公式 | 环境中可见的变换 |
| 抽象过程 | 粒子/物体的具象流动 |

**错误方式**：角色指着黑板解释
**正确方式**：角色使用"概念可视化器"道具，走进隐喻空间

### 视觉隐喻示例

| 概念 | 错误（说教式对话） | 正确（视觉隐喻） |
|---------|---------------------|------------------------|
| 注意力机制 | 角色指着黑板上的公式 | "注意力手电筒"道具在暗室中照亮关键词 |
| 梯度下降 | "算法最小化损失" | 角色骑着球沿山谷滚下 |
| 神经网络 | 带箭头的图表 | 由发光生物组成的活网络传递信息 |
| 过拟合 | "模型记住了数据" | 角色穿着只适合一个特定姿势的衣服 |

### 角色设定（必需）

**默认：使用哆啦A梦角色**，除非用户明确指定 `--characters` 或在 EXTEND.md 中有角色预设。

| 角色 | 默认角色 | 视觉 | 特点 |
|------|-------------------|--------|--------|
| 学生（角色 A） | 大雄（Nobita） | 男孩，10 岁，圆框眼镜，黑发，黄色上衣，藏蓝短裤 | 困惑，提出基础但关键的问题，代表读者 |
| 导师（角色 B） | 哆啦A梦（Doraemon） | 蓝色机器猫，白色肚子，四维口袋，红鼻子，金色铃铛 | 知识渊博，耐心，用道具作为技术隐喻 |
| 挑战（角色 C） | 胖虎（Gian） | 壮实男孩，小眼睛，橙色上衣 | 代表误解，或数据中的"噪声" |
| 支援（角色 D） | 静香（Shizuka） | 可爱女孩，黑色短发，粉色连衣裙 | 提出澄清性问题，提供不同视角 |

**重要**：这些哆啦A梦角色是 ohmsha 预设的默认角色。除非用户另有要求，否则使用这些确切角色生成角色定义。

使用自定义角色：`--characters "Student:小明,Mentor:教授"` 或在 EXTEND.md 中定义。

### 页面标题规范

每一页必须有叙事性标题（非章节标题）：

**错误**："第 1 章：Transformers 简介"
**正确**："大雄听不懂任何人说话的那一天"

### 道具揭秘模式

引入概念时：

1. 学生用视觉符号表达困惑（？、旋转眼）
2. 导师戏剧性地拿出道具，配合闪光效果
3. 道具名称以粗体公布并附带解释
4. 开始演示 - 学生进入隐喻空间

### 结尾要求

最后一页必须包含：

1. 学生展示理解（应用概念）
2. 回调开头问题（现已解决）
3. 导师满意的表情
4. 可选：暗示下一个主题

### 禁止说教式对话规则

**关键**：角色必须在做事情，而不仅仅是解释。

每个分格应展示：
- 正在执行的动作
- 正在演示的隐喻
- 角色与概念空间的互动
- 不要：两个角色面对面说话

### 特殊视觉元素

| 元素 | 用途 |
|---------|-------|
| 道具揭秘 | 戏剧性揭幕配闪光效果 |
| 概念空间 | 圆角边框，发光边缘表示"想象模式" |
| 信息展示 | 全息 UI 风格展示技术细节 |
| 恍然大悟时刻 | 放射线，光爆效果 |
| 困惑 | 旋转眼，头顶漂浮问号 |

## 质量标记

- ✓ 每个概念都是视觉隐喻
- ✓ 角色在做事情，而非只是说话
- ✓ 清晰的师生互动
- ✓ 道具和工具驱动解说
- ✓ 富有表现力的漫画式情感
- ✓ 通过视觉设计而非文字堆砌实现信息密度
- ✓ 叙事性页面标题
